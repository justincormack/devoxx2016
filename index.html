Docker Security for Developers

No Silos

No silos...

  • DevSecOps
  • Teams working together to deliver quality software
  • Security as an ongoing process not a checkbox

Organization first

  • Embed security people in your teams. They need to understand what your product is doing too.
  • Understand the threat model from the start, and the main risks
  • Postmortems when things go wrong, not to blame, to improve

Automation and tooling second

  • Once the team structure is in place, start to find tools
  • Tools that generate hundreds of pages of PDFs are no use
  • You want to validate behaviour in test and in production

SHIP

Docker Content Trust

  • Sign images at point of authorship (using Notary)
  • Removes implicit trust of storage service and network
  • Guarantee integrity of your images when pulled

Image Scanning

  • Looks inside your images and finds libraries you use
  • Checks if there are known security issues
  • Service available at cloud.docker.com

Known issues are a major problem developers can fix

  • PCI Data Breach Investigation Report (DBIR) studying 85,000 data breaches in 2014 said that ten known CVEs accounted for 97% of exploits.
  • Eight of the ten were over ten years old.
  • The typical organization uses 18,614 different software components. 7.5% had known vulnerabilities.

RUN

Least Privilege

  • Do not give your code more access than it needs
  • Guides all configuration in a secure environment
  • Any attacker or mistakes does not get to access more than the minimum, contained.

Least Privilege

  • You need to understand what your application does, and what else it needs access to
  • Microservices make this easier: one service, one function
  • Small simple services can be more secure because you understand everything about them

Least Privilege

  • Containers are about least privilege!
  • They restrict the files available to containers
  • They restrict network access, to just published ports

Least Privilege

  • Docker filters some system calls that are high risk, with the default seccomp filter
  • Docker applies Apparmor or SELinux profiles if your system supports them to restrict escalation
  • Linux capabilities are restricted can be added with --cap-add in rare cases they are needed.
  • Avoid using --privileged or --cap-add SYS_ADMIN options, these give high privileges.
“Gartner asserts that applications deployed in containers are more secure than applications deployed on the bare OS.”

Secrets

  • Don't commit secret keys into your source repository!
  • Don't use environment variables for secrets
  • Can use key storage mechanisms like Vault
  • Docker 1.13 is going to include built in secrets management
